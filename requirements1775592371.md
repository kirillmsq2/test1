## Глоссарий

| Термин / Сокращение | Описание |
|---------------------|----------|
| **ERIP** | Единый Расчётный Информационный Портал – внешняя система, предоставляющая сервисы публикации и контроля QR‑платежей. |
| **RTP** | Real‑Time Payments – директория в ERIP, где размещаются объекты (Envoys) и их статусы. |
| **SMP** | Система Межбанковских Переводов – отвечает за денежный поток между банками‑бенефициарами. |
| **QR‑code** | Двумерный штрих‑код, используемый клиентом для инициирования оплаты. |
| **Static QR** | QR‑код, заранее размещённый в точке продаж (наклейка, печатный материал). |
| **Dynamic QR** | QR‑код, формируемый «на лету» на основе текущей корзины покупателя. |
| **Envoy** | Объект платежа в директории RTP (содержит сумму, получателя, идентификатор). |
| **risk‑indicator.ts** | Файл/поле, содержащие показатель риска транзакции; рассчитывается анти‑фрод системой. |
| **anti‑fraud** | Система обнаружения мошенничества, в текущей реализации поддерживает только физ. лиц. |
| **SD‑BO** | Система Дистанционного Банковского Обслуживания (модуль, требующий доработки). |
| **Белокасса** | Продукт банка, объединяющий два поставщика ПО: «Кассал‑ПО» (цифровой) и «Терминальное ПО». |
| **Кассал‑ПО** | Программное обеспечение для работы с кассами, реализующее функции приёма карт и наличных. |
| **Терминальное ПО** | Программное обеспечение, установленное в терминалах (POS‑терминалы). |
| **API** | Программный интерфейс, через который банк будет взаимодействовать с ERIP. |
| **Refund** | Возврат средств клиенту после подтверждённого платежа. |
| **BR_XXX** | Идентификатор бизнес‑требования. |
| **FR_X.X** | Идентификатор функционального требования. |

## 2. Функциональные требования

### 2.1. Схемы процессов и вариантов использования

#### 2.1.1. Регистрация клиента/терминала в RTP (Use Case 1)

1. **Бизнес‑операция** – заключён договор с клиентом (или выбранный терминал).  
2. **Система «Белокасса»** формирует запрос `POST /clients` к API ERIP с данными клиента (ИНН, договор, тип).  
3. **ERIP** сохраняет запись, возвращает уникальный `clientId`.  
4. **«Белокасса»** сохраняет `clientId` в своей базе и помечает клиента как «готовый к QR‑оплате».  

#### 2.1.2. Генерация QR‑кода (Use Case 2)

*Вариант A – статический QR*  
1. Точка продаж получает от «Белокассы» заранее сгенерированный QR‑код (фиксированный `invoiceId`).  
2. QR размещается на кассе (наклейка, экран).  

*Вариант B – динамический QR*  
1. Клиент формирует корзину в мобильном приложении банка.  
2. «Белокасса» формирует запрос `POST /envoys` к ERIP с суммой, получателем, `clientId`.  
3. ERIP возвращает `envoyId` и URL/данные QR‑кода.  
4. QR отображается на экране терминала или в приложении клиента.  

#### 2.1.3. Обработка оплаты (Use Case 3)

1. Клиент сканирует QR‑код, приложение банка отправляет запрос `GET /envoys/{envoyId}` к ERIP.  
2. ERIP запрашивает у «Белокассы» дополнительные данные (risk‑indicator, статус счёта).  
3. **Anti‑fraud** рассчитывает `risk‑indicator.ts` и возвращает результат.  
4. При положительном результате ERIP инициирует денежный перевод через SMP.  
5. После подтверждения SMP, ERIP меняет статус Envoy → *Paid* и отправляет callback в «Белокассу».  
6. «Белокасса» обновляет статус в своей системе и информирует точку продаж.  

#### 2.1.4. Возврат средств (Use Case 4)

1. Оператор точки продаж инициирует возврат в «Белокассе».  
2. «Белокасса» формирует запрос `POST /refunds` к ERIP с `envoyId`.  
3. ERIP проверяет статус, инициирует обратный перевод через SMP.  
4. После завершения ERIP меняет статус Envoy → *Refunded* и отправляет callback.  
5. «Белокасса» закрывает операцию и уведомляет клиента.

### 2.4. Границы решения

**В границы решения входит:**
- Регистрация клиентов и терминалов в директории RTP ERIP.  
- Генерация статических и динамических QR‑кодов.  
- Обработка платежей и возвратов через ERIP + SMP.  
- Расчёт risk‑indicator с доработкой anti‑fraud.  
- Интеграция с существующим продуктом «Белокасса» (UI‑вывод QR, API‑взаимодействие).  

**Вне границ решения (out of scope):**
- Полная переработка кассового ПО поставщиков (вендоры отвечают за свои UI‑изменения).  
- Модернизация ядра anti‑fraud, кроме добавления расчёта risk‑indicator для юридических лиц.  
- Разработка новых каналов оплаты (NFC, Apple Pay и пр.).  
- Регулятивные изменения, не связанные с QR‑оплатой.
