**Данных достаточно для построения sequence‑диаграммы.**  
Ниже – основная (основная) последовательность взаимодействий, а также альтернативные варианты (статический / динамический QR, внутрибанковский / межбанковский платёж).

---

```plantuml
@startuml
title Платёж по QR‑коду через сервис ERIP

'--- Участники -------------------------------------------------
actor "Клиент (мобильное приложение банка)" as Client
participant "Точка продаж / Касса (PO‑система)" as POS
boundary "Сервис ERIP (платёжный сервис)" as ERIP
participant "Анти‑фрод система" as AntiFraud
boundary "Система межбанковских платежей (SMP)" as SMP
database "Директория RTP (хранилище Envoy)" as RTP

'--- Регистрация терминалов/клиентов (один раз) -----------------
group Регистрация терминала/клиента
    note left of POS : Регистрация в директории RTP\nчерез API ERIP
    POS -> ERIP : RegisterTerminal(данные терминала)
    ERIP -> RTP : Сохранить запись о терминале
    RTP --> ERIP : OK
    ERIP --> POS : Регистрация завершена
end

'=== Основной сценарий: оплата по динамическому QR ===
alt Динамический QR (корзина известна)
    POS -> ERIP : CreateEnvoy(корзина, сумма)
    ERIP -> RTP : Сохранить Envoy, присвоить UID
    RTP --> ERIP : UID
    ERIP --> POS : QR‑code(UID)   'генерация QR
else Статический QR (предзаписан)
    note right of POS : QR‑code уже размещён на кассе
end

'--- Клиент сканирует QR и инициирует оплату ---------------
Client -> POS : Сканировать QR (UID)
Client -> ERIP : PayEnvoy(UID, данные клиента)
ERIP -> RTP : Запрос Envoy по UID
RTP --> ERIP : Envoy (сумма, детали)

'--- Проверка риска -----------------------------------------
alt Платёж от физ. лица
    ERIP -> AntiFraud : GetRiskIndicator(данные клиента)
    AntiFraud --> ERIP : risk‑indicator
else Платёж от юрид. лица
    note right of ERIP : Risk‑indicator пока не определён\n(требуется доработка)
end

'--- Перевод средств ----------------------------------------
alt Межбанковский платёж (банк‑бенефициар ≠ банк‑плательщик)
    ERIP -> SMP : TransferFunds(сумма, UID)
    SMP --> ERIP : OK (перевод выполнен)
else Внутрибанковский платёж (один банк)
    ERIP -> ERIP : InternalTransfer(сумма, UID)
    ERIP --> ERIP : OK
end

'--- Информирование о статусе -------------------------------
ERIP -> RTP : UpdateEnvoyStatus(UID, «оплачен»)
RTP --> ERIP : OK
ERIP -> POS : NotifyPayment(UID, статус)
POS --> Client : Платёж подтверждён (чек/уведомление)

'=== Альтернативный сценарий: возврат =======================
alt Возврат товара
    POS -> ERIP : RefundRequest(UID)
    ERIP -> SMP : ReverseTransfer(UID)
    SMP --> ERIP : OK
    ERIP -> RTP : UpdateEnvoyStatus(UID, «возвращён»)
    RTP --> ERIP : OK
    ERIP -> POS : NotifyRefund(UID)
    POS --> Client : Возврат подтверждён
end

@enduml
```

### Допущения
* Точная названия API‑методов (например `CreateEnvoy`, `PayEnvoy`) выбраны условно, исходя из описания функций.
* Внутрибанковский перевод реализован как внутренний вызов внутри ERIP – в реальном решении может быть отдельный сервис.
* Для статического QR‑кода процесс создания Envoy пропускается – QR уже существует в POS.
* В случае юридических лиц риск‑индикатор пока не реализован (требуется доработка анти‑фрода).

Если нужны более детальные сообщения (например, конкретные названия методов, форматы запросов/ответов) или отдельные сценарии (например, только регистрация без оплаты), уточните эти детали – диаграмму можно уточнить.
