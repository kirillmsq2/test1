**Краткий анализ встречи**

1. Тема – разработка сервиса QR‑платежей, который будет использоваться в точках продаж банка‑бенефициара и взаимодействовать с внешней системой ERIP.  
2. Ключевые участники/системы – Плательщик (мобильное приложение банка), Точка продаж (кассовое/терминальное ПО), Сервис‑провайдер ERIP (внешний), Директория RTP, Система анти‑фрода, Межбанковская система расчётов SMP, Внутренний платежный сервис ERIP (middleware), База данных/хранилище запросов (Envoys).  
3. Описанный процесс – Плательщик сканирует QR‑код (статический или динамический), генерируется запрос Envoy через middleware‑ERIP, сохраняется в директории RTP, мобильное приложение банка инициирует оплату, банк‑плательщик запрашивает детали у RTP, анти‑фрод рассчитывает `risk‑indicator.ts`, ERIP возвращает статус, деньги переводятся через SMP (межбанковский сценарий) или напрямую через BIS (внутрибанковский), статус оплаты возвращается в ERIP и далее в точку продаж.

| № | Что уточнено/дополнено | Почему |
|---|------------------------|--------|
|1|`Envoy` – это объект платежного запроса, содержащий сумму, корзину и уникальный идентификатор. | В тексте упоминается «формируем Envoys». |
|2|Для статического QR‑кода QR генерируется заранее и хранится в терминале; для динамического – генерируется on‑demand через middleware‑ERIP. | Описаны два варианта QR. |
|3|`risk‑indicator.ts` рассчитывается внешней системой анти‑фрода и возвращается в виде отдельного сообщения. | В тексте сказано, что анти‑фрод будет доработан. |
|4|Система SMP используется только при межбанковском переводе; при внутрибанковском переводе используется BIS (direct settlement). | Указано разделение денежных и информационных потоков. |

|6|Ответы от ERIP к банку‑плательщику и к точке продаж передаются через `-->` (пунктир). | Стандартный стиль ответа. |

**PlantUML‑диаграмма**

```plantuml
@startuml
title QR‑платёж через ERIP (основной сценарий)

' Участники
actor "Плательщик (моб. приложение банка)" as Customer
participant "Точка продаж (Кассовое/Терминальное ПО)" as POS
participant "Middleware ERIP (платёжный сервис)" as Middleware
boundary "Директория RTP (ERIP)" as RTP
participant "Анти‑фрод система" as AntiFraud
actor "Банк‑плательщик" as PayerBank
actor "Банк‑бенефициар" as BenefBank
boundary "SMP (межбанковский расчёт)" as SMP
database "Хранилище Envoy" as EnvoyDB

' == Шаг 1: Формирование QR‑кода ==
group QR‑code generation
    POS -> POS : Вывести статический QR (если используется)
    alt Динамический QR
        POS -> Middleware : Запрос генерации QR (корзина, сумма)
        Middleware -> RTP : Публикация Envoy (async) >>
        RTP --> Middleware : ID Envoy
        Middleware -> POS : QR‑code (URL с ID) ->> 
    end
end

' == Шаг 2: Сканирование QR и инициирование оплаты ==
Customer -> POS : Сканировать QR (получить ID Envoy)
Customer -> Middleware : Запрос оплаты (ID Envoy)
Middleware -> RTP : Запрос деталей Envoy
RTP --> Middleware : Данные Envoy (сумма, получатель)

' == Шаг 3: Проверка риска ==
Middleware -> AntiFraud : Запрос risk‑indicator.ts
AntiFraud --> Middleware : risk‑indicator.ts

' == Шаг 4: Авторизация и перевод средств ==
alt Межбанковский сценарий
    Middleware -> PayerBank : Запрос авторизации платежа
    PayerBank -> SMP : Перевод средств (SMP)
    SMP --> PayerBank : Подтверждение
else Внутрибанковский сценарий
    Middleware -> PayerBank : Перевод через BIS (direct)
    PayerBank --> Middleware : Подтверждение
end

' == Шаг 5: Обновление статуса ==
Middleware -> RTP : Обновить статус Envoy = «Оплачен»
RTP --> Middleware : ACK
Middleware -> POS : Уведомление об оплате (status)
POS --> Customer : Выдача чека / подтверждение

note over Middleware,RTP
    Информационный поток (запрос/ответ) полностью проходит через ERIP.
end note

note over SMP
    Денежный поток (перевод средств) проходит через SMP только в межбанковском случае.
end note

@enduml
```
