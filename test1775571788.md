## 1. Бизнес‑требования  

### 1.1. Цели / задачи / назначение  

- **Обеспечить возможность оплаты товаров/услуг через QR‑код** как статический (заранее размещённый) и динамический (генерируемый на лету).  
- **Автоматизировать регистрацию клиентов и терминалов** в директории RTP, чтобы каждый клиент‑плательщик мог использовать QR‑оплату без участия сторонних сервисов.  
- **Поддержать процесс возврата средств** через тот же канал (ERIP).  
- **Минимизировать доработки в существующей системе СДБО/Белокасса**, используя уже реализованные функции QR‑оплаты в мобильном приложении банка.  
- **Обеспечить согласованность информационных и денежных потоков**: информационный поток – через ERIP, денежный – через СМП (для межбанковских операций) либо через BIS (для внутрибанковских).  

| ID | Бизнес‑требование | Цель и результат |

| BR_001 | Реализация оплаты QR‑кодом (статический/динамический) | Увеличение количества безналичных транзакций, снижение нагрузки на кассовый процесс |
| BR_002 | Регистрация клиентов и терминалов в RTP | Полный контроль над клиентской базой, возможность аналитики и мониторинга |
| BR_003 | Поддержка возврата средств через ERIP | Сокращение времени возврата, повышение клиентского удовлетворения |
| BR_004 | Минимальные изменения в СДБО/Белокасса | Сокращение сроков реализации, снижение стоимости проекта |
| BR_005 | Разделение информационных и денежных потоков | Обеспечение соответствия регулятивным требованиям и упрощение аудита |

### 1.3. Контекстная диаграмма участвующих систем  

- **Клиент (физическое лицо)** – сканирует QR‑код, инициирует оплату через мобильное приложение банка.  
- **Мобильное приложение банка** – формирует запрос к ERIP (если динамический QR) и получает статус оплаты.  
- **Система Белокасса (POS‑PO + терминальное ПО)** – отображает статический QR‑код, передаёт данные о транзакции в ERIP.  
- **Платёжный сервис ERIP** – хранит и публикует Envoy в директории RTP, проверяет статус, взаимодействует с анти‑фродом, возвращает результат банку‑плательщику.  
- **Директория RTP** – реестр всех запросов и их статусов, обеспечивает уникальные идентификаторы.  
- **Система анти‑фрод** – рассчитывает `risk‑indicator.ts` и передаёт в ERIP.  
- **Система межбанковских платежей (СМП)** – осуществляет денежный перевод между банком‑плательщиком и банком‑бенефициаром (для межбанковских сценариев).  
- **BIS / внутренний процессинг** – используется при внутрибанковском переводе (без СМП).  

| Элемент / Система | Описание участия в процессе | Потенциальные доработки |
|-------------------|----------------------------|------------------------|
| Клиент | Инициирует оплату, сканирует QR | — |
| Мобильное приложение банка | Формирует запрос к ERIP, получает статус | Возможна доработка UI/UX для отображения QR |
| Белокасса (POS‑PO) | Выводит статический QR, передаёт данные о покупке | Интеграция с API ERIP для генерации динамического QR |
| ERIP | Управление Envoy, хранение, проверка статуса, возврат | Добавление методов регистрации клиентов/терминалов |
| RTP‑директория | Хранилище запросов, генерация уникальных ID | — |
| Анти‑фрод | Расчёт risk‑indicator.ts | Доработка для поддержки юридических лиц |

| BIS | Внутрибанковский перевод | — |

## 2. Функциональные требования  

### 2.1. Схемы процессов и вариантов использования  

#### 2.1.1. Оплата через QR‑код (динамический)  

1. **Клиент** формирует корзину в точке продаж.  
2. Точка продаж **запрашивает у ERIP** генерацию динамического QR‑кода (передаёт сумму, описание, идентификатор терминала).  
3. **ERIP** создаёт Envoy, сохраняет его в RTP, возвращает URL/изображение QR‑кода.  
4. **Точка продаж** отображает QR‑код на экране/терминале.  
5. **Клиент** сканирует QR‑код мобильным приложением банка.  
6. Приложение **отправляет запрос** в ERIP о подтверждении оплаты.  
7. **ERIP** проверяет статус Envoy, запрашивает `risk‑indicator.ts` у анти‑фрода, при положительном результате – списывает средства.  
8. **ERIP** отправляет подтверждение в **банк‑плательщик**, который через СМП переводит деньги банку‑бенефициару.  
9. **Банк‑бенефициар** информирует точку продаж о завершении операции.  

#### 2.1.2. Оплата через QR‑код (статический)  

1. На кассе размещён **статический QR‑код** (привязан к конкретному мерчанту).  
2. **Клиент** сканирует код, открывается приложение банка с предзаполненными данными (сумма фиксирована).  
3. Дальнейшие шаги аналогичны пункту 6‑9 динамического сценария.  

#### 2.1.3. Регистрация клиента/терминала в RTP  

1. **Банк** формирует список клиентов (по договорам) и терминалов, которые должны участвовать в QR‑платежах.  
2. Через **API ERIP** отправляется запрос `RegisterClient`/`RegisterTerminal` с необходимыми атрибутами (идентификатор, тип, статус).  
3. **ERIP** сохраняет запись в RTP, возвращает подтверждение и уникальный `clientId`/`terminalId`.  
4. При необходимости **анти‑фрод** получает данные о клиенте для дальнейшего расчёта риска.  

#### 2.1.4. Возврат средств  

1. Точка продаж инициирует запрос **Refund** к ERIP, указывая `envoyId`.  
2. **ERIP** проверяет статус оригинального платежа, инициирует обратный денежный перевод через СМП (или BIS).  
3. После подтверждения возврата **ERIP** обновляет статус в RTP и уведомляет банк‑плательщик.

### 2.4. Границы решения  

**В границы решения входит:**  

- Интеграция с сервисом ERIP (генерация QR, регистрация, возврат).  
- Реализация API‑вызовов регистрации клиентов и терминалов.  
- Доработка UI мобильного приложения банка для отображения QR‑кода и статуса.  
- Минимальные изменения в POS‑PO и терминальном ПО (вывод QR‑кода).  
- Подключение к системе анти‑фрод для расчёта `risk‑indicator.ts`.  

**Вне границ решения (out of scope):**  

- Полная переработка кассового программного обеспечения (POS‑PO) – только вывод QR‑кода.  
- Расширение функционала анти‑фрод за пределами расчёта `risk‑indicator.ts`.  
- Интеграция с другими платёжными шлюзами, не входящими в ERIP.  
- Поддержка QR‑оплаты в офлайн‑режиме без доступа к сети.
